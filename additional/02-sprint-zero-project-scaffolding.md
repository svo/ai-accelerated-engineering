# Sprint Zero & Project Scaffolding

## Overview

Sprint zero is the most leveraged moment in a project's lifecycle - decisions made here compound across every subsequent sprint. AI can dramatically compress the time from "new repository" to "production-ready service" by generating project scaffolds that bake in architectural constraints, testing infrastructure, deployment pipelines, and documentation from the start. Three production-ready templates demonstrate the approach: [**python-sprint-zero**](https://github.com/svo/python-sprint-zero) (FastAPI, pytest, Lagom DI, Vagrant/Docker/Ansible/Packer), [**typescript-sprint-zero**](https://github.com/svo/typescript-sprint-zero) (Node.js, Vitest, tsyringe), and [**www-qual-is**](https://github.com/svo/www-qual-is) (Next.js 15, React 19, Tailwind CSS, Vitest, Playwright) - all enforcing hexagonal architecture with 100% test coverage from day one. These templates have been applied at scale across multi-service platforms with multiple microservices and Next.js frontends.

## Approaches

### Template-Driven Generation with AI Customisation

Start from a proven service template and use AI to adapt it to your specific domain. Rather than manually renaming modules and rewriting configurations, describe the service's purpose and let the AI tailor the scaffold - domain models, API routes, dependency injection bindings, and test fixtures included. [**python-sprint-zero**](https://github.com/svo/python-sprint-zero) provides the primary microservice template (FastAPI with Lagom DI, structured as domain/application/infrastructure/interface layers), while [**www-qual-is**](https://github.com/svo/www-qual-is) serves as the web application template (Next.js 15 with hexagonal architecture, Tailwind CSS, and markdown-based content). Claude Code skills embedded in these templates handle the customisation.

### Architectural Constraint Enforcement from Day One

Encode hexagonal architecture boundaries into the project structure and tooling before a single line of business logic is written. Automated validation ensures that infrastructure never leaks into domain code, even as the team grows. [**python-sprint-zero**](https://github.com/svo/python-sprint-zero) uses [pytest-archon](https://github.com/jwbargsten/pytest-archon) to enforce layer boundaries, while [**www-qual-is**](https://github.com/svo/www-qual-is) and [**typescript-sprint-zero**](https://github.com/svo/typescript-sprint-zero) ship with `.dependency-cruiser.js` configurations - domain must not import infrastructure, and violations fail the build. AI can generate these rules from an architectural description.

### Framework-Driven API Contracts

OpenAPI specifications are auto-generated by the frameworks (e.g., FastAPI). AI assists by defining API contracts from natural language descriptions of service boundaries, ensuring frontend and backend teams can work in parallel from sprint one.

### Consistent Multi-Service Scaffolding

When building a platform with multiple microservices, AI maintains consistency across services - shared patterns for health checks, authentication middleware, error handling, and logging. This has been demonstrated across platforms with multiple services sharing consistent patterns for OpenAPI auto-generation, Basic Auth, health checks, and database integration - each scaffolded from the same architectural template. Larger platforms extend this further with many microservices and a Next.js UI, all following hexagonal architecture with 100% test coverage.

### Infrastructure-as-Code Generation

Generate Ansible playbooks, Dockerfiles, and CI/CD pipelines alongside application code. [**python-sprint-zero**](https://github.com/svo/python-sprint-zero) includes Vagrant for local development, Ansible for server provisioning, Packer for machine images, and Docker for containerisation - all generated alongside the application code. A sprint zero that includes deployment infrastructure means the first feature can ship to a real environment, not just localhost.
