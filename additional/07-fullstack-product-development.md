# Full-Stack Product Development

## Overview

Building a multi-service product requires coordinating changes across frontends, backends, APIs, and infrastructure. A single AI agent working across all of these lacks the focused context to do any one well. The more effective pattern is an orchestrating agent that holds the macro context - architecture, service boundaries, data contracts - and generates targeted prompts for specialist agents that each work within a single component. This guide explores how to structure that orchestration, drawing from production platforms built with Python microservices and Next.js 15/React 19 frontends - platforms where multiple services must evolve in coordination, each with its own specialist agent scoped to a single codebase.

## Approaches

### Orchestrator-Specialist Architecture

Define an orchestrating agent that understands the full product: its service map, API contracts, shared types, and architectural constraints. When a feature requires changes across multiple components, the orchestrator decomposes the work into targeted prompts - one per frontend, microservice, or module - each scoped with just the context that agent needs. The specialists execute; the orchestrator aligns.

### Feature Decomposition Across Services

A product feature rarely lives in one place. "Add email notifications for a key event" might touch a scheduling service, a notification service, an identity service (for contact details), and the frontend (for user preferences). The orchestrator analyses the feature, identifies which services are affected, determines the order of changes (API contract first, then backend, then frontend), and generates a prompt for each specialist agent with the relevant contracts and constraints.

### Framework-Driven Contracts

The orchestrator works with API contracts auto-generated by the frameworks (e.g., FastAPI's OpenAPI specs). Each specialist agent receives the relevant contracts as part of its prompt, ensuring all components agree on data shapes and interfaces.

### Consistency Enforcement Across Components

After specialist agents produce their output, the orchestrator reviews for consistency - do error codes match across services? Are shared types aligned? Do naming conventions hold? This review step catches the drift that naturally occurs when multiple agents (or developers) work in parallel.

### Progressive Build-Out

Start with the orchestrator generating a minimal version of each component - in-memory storage, basic auth, stub integrations. This pattern is applied across platforms: services start with in-memory repositories and Basic Auth, then progressively swap in production implementations (databases, managed auth, payment providers) as the domain model stabilises. Each iteration is scoped and targeted, not a full rewrite.

### Scoped Agent Context

Each specialist agent operates within a single codebase and receives only the context it needs - the relevant API contracts, architectural rules for that layer, and the specific change required. This scoping prevents agents from being overwhelmed by irrelevant context and produces more focused, higher-quality output.

## Tooling

| Tool | Purpose |
|------|---------|
| **[Claude Code](https://docs.anthropic.com/en/docs/claude-code)** | Specialist agents working within individual Python microservices or Next.js frontends |
| **[FastAPI](https://fastapi.tiangolo.com)** | Python web framework for microservice APIs |
| **[Lagom](https://lagom-di.readthedocs.io) / [tsyringe](https://github.com/microsoft/tsyringe)** | Dependency injection for Python and TypeScript services respectively |
| **[Next.js 15](https://nextjs.org) / [React 19](https://react.dev)** | Web application framework for frontends |
| **[OpenAPI](https://www.openapis.org)** | Contract definitions that the orchestrator generates and specialists consume (auto-generated by services) |
| **[dependency-cruiser](https://github.com/sverweij/dependency-cruiser)** | Validates architectural boundaries within each service and across the platform |
| **[pytest-archon](https://github.com/jwbargsten/pytest-archon)** | Validates architectural boundaries in Python projects |
| **[pytest](https://docs.pytest.org) / [Jest](https://jestjs.io) / [Vitest](https://vitest.dev)** | Testing frameworks verifying each specialist's output (100% coverage enforced) |
| **[OpenAPI](https://www.openapis.org) / [Prism](https://github.com/stoplightio/prism) CDCT** | Consumer-driven contract testing across service boundaries |
| **[Docker](https://www.docker.com) / [Vagrant](https://www.vagrantup.com)** | Running multi-service environments locally to validate end-to-end behaviour |
| **[Terraform](https://www.terraform.io) / [Terragrunt](https://terragrunt.gruntwork.io)** | Infrastructure provisioning for the full platform |

## Key Questions

- How do you currently coordinate changes that span multiple services - and where does that process break down?
- What context does a developer need when working on a single service, and how much of it could an orchestrator provide automatically?
- Where would an orchestrator-specialist pattern save the most time in your current product development workflow?
